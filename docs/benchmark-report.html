Fork: N sandboxes from one snapshot

Fan-out resumes N identity-neutral clones from a single snapshot — each with a fresh IP/tap/port and a copy-on-write rootfs, sharing snapshot memory via the page cache. The fixed batch overhead amortizes: per-clone cost collapses from 1.7 s at N=1 to ~72 ms at N=48. (Post-GARP, a single clone ≈ hot create ≈ 250 ms.)

Burst behavior

500-sandbox churn (create → exec → kill, concurrency 96) against 3 workers × 24 slots: 499/500 succeeded in 72 s — 6.9 creates/s sustained — with client retry+backoff; zero pool-exhaustion errors after the gateway's reserve-at-pick fix, clean 503s under overload. Sustained overload (500 held alive) ramped the autoscaler 3→8 hosts over ~450 s. Full write-up: 500-sandbox burst benchmark.

Honest caveats

  • Network position. Our end-to-end numbers ride a tailnet to our own gateway; hosted numbers include WAN + the provider's API edge. Self-hosting earns that advantage, but it's an advantage of position, not implementation.
  • Hardware. Our fleet is nested-KVM GCP VMs; bare metal measured faster (cold boot ~2.2 s vs 3.5 s). These aren't best-case numbers.
  • Warm strategies everywhere. Hosted providers keep warm pools; our hot path is a per-host golden snapshot built at startup. Only the cold-boot rows are cold-metal numbers.
  • Claims vs measurements. Anything labeled claim is vendor marketing or docs without published methodology. Where an independent measurement exists, it disagrees with the claim in every case except Modal's.
