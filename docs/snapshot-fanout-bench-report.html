01 · Latency

Time to one usable sandbox

Cold boot copies a rootfs, boots the kernel, runs init, and waits for the agent. Restore loads a snapshot's memory + device state and resumes — skipping all of that. Both numbers are "time until you can exec in the box," measured over 25 iterations on a single host.

Cold boot Snapshot restore
Methodmeanp50p90minmax
16.3×

Restore is 16.3× faster at the median (212 ms vs 3463 ms). Snapshot capture itself took ~1.99 s, paid once. The whole win comes from skipping kernel boot + init + agent startup — the agent is already running in the restored memory.

02 · Fan-out scaling

One snapshot → N live clones, in one call

Fan-out restores many identity-neutral clones from a single snapshot at once — each gets a fresh IP/tap/port and its own copy-on-write rootfs. Because a fixed reidentification step is shared across the whole batch, the amortized cost per clone collapses as N grows.

N clonesbatch (ms)per-clone (ms)usable
12.7×

At N=64, fan-out brings up all clones in 5.7 s vs 72.8 s for 64 concurrent cold boots — 12.7× faster. Per-clone cost bottoms out around 72 ms at N=48; near the 64/host pool ceiling, resource contention nudges it back up.

Note: the ~1.5 s fixed floor at small N is a deliberate reidentification wait before bridging each clone's network. Replacing it with a readiness signal is the top open optimization.

03 · Fleet throughput

Create + run a workload across both hosts, via the gateway

End-to-end load test: the gateway places N sandboxes across the two hosts, then a workload runs in every one concurrently. All creates succeeded through 64; at 128 (both hosts packed to their 64-slot ceiling), the CPU-bound workload oversubscribes the 8-core hosts 16:1 and the run degrades.

Read: create latency stays flat (~9 s p50 under concurrency 12) across all sizes — the fleet provisions reliably. The 128 result is a workload-saturation ceiling, not a provisioning failure: 128/128 were created; execs dropped once every core was contended.

04 · Memory density

What 64 sandboxes actually cost in RAM

Fan-out clones mmap the same snapshot memory file, so their pages are shared in the page cache (copy-on-write until written). PSS (proportional set size) counts shared pages once, so it reflects the true footprint; RSS counts them in every process. Cold boots share nothing.

64 fan-out clones occupy 925 MB of real memory (PSS) versus 10.1 GB for 64 cold boots — ~14 MB/clone vs ~157 MB/clone. Shared snapshot memory is the density lever; it's what makes packing dozens of clones per host practical.
